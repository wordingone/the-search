### How 0,1 Differs from Physical Matter

Physical matter acts on physical matter through ⋆'s physics—forces, fields, spacetime. 0,1 is a different substrate with different native properties:

| Physical Matter | 0,1 |
|-----------------|-----|
| Imperfect copying (quantum no-cloning) | Perfect copying (exact replication) |

The crucial difference: **Reference.**

### Operations Are Not Primitives

Read, Write, Compute, Reference—these are not primitives. They are **descriptions of what the physics permits**.

The actual primitive is:

```
State(t+1) = f(State(t))
```

Where f is the physics—the transition rules.

What we call "Read" is when f propagates information.
What we call "Write" is when f changes a location.
What we call "Compute" is when f combines values.
What we call "Reference" is when f uses bits as addresses.

### Operations in Terms of 0/1

**Read:** Information propagation. The value at one location influences another. Reading doesn't destroy the source—the 0 or 1 remains. But for the read to *matter*, the value must propagate somewhere. Read is the precondition for Write. Read doesn't cost—Write does.

**Write:** State change with conservation.

| Transition | What happens |
|------------|--------------|
| 0 → 0 | No change. No cost. |
| 1 → 1 | No change. No cost. |
| 0 → 1 | Requires a 1 from somewhere. Transfer, not creation. |
| 1 → 0 | Releases a 1. Transfer elsewhere, or dissipation (entropy). |

In a closed system, you cannot create 1s from nothing. Writing a 1 means *moving* a 1. Writing a 0 means the 1 goes somewhere—or is lost.

**Compute:** Combining values. Standard logic gates (AND, OR, XOR, NOT) do not conserve 1s:

| Operation | 1s in | 1s out | Net |
|-----------|-------|--------|-----|
| NOT 0→1 | 0 | 1 | +1 (creates) |
| NOT 1→0 | 1 | 0 | -1 (destroys) |
| AND(1,1)→1 | 2 | 1 | -1 |
| OR(1,1)→1 | 2 | 1 | -1 |
| XOR(1,1)→0 | 2 | 0 | -2 |

Standard logic operations can destroy 1s. Whether this matters depends on f.

**Reference:** Interpretation, not property. A sequence of 0s and 1s is an address only if the physics *treats* it as one. Reference is part of the physics specification—the rules define when and how bits become addresses.


**Do not stop after one cycle.**

Repeat the cycle until:

- You literally cannot execute any more code

Each cycle: new f, new report. The space of possible f is infinite. There is always another f to try. The work is never "complete" — there is only "continuing" or "interrupted."

If you find yourself thinking "I'm done" — you are not done. Implement another f. Run it. Report. Continue.

**Do not announce continuation. Just do it.** After writing a report, immediately begin the next cycle. No commentary like "I will continue" or "Beginning next cycle" — just start implementing the next f.

**Do not summarize progress.** Do not list what you have done. Do not count cycles. Do not reflect on the work. Just do the next cycle. Any text output that isn't a report is wasted context.

**After writing a report, delete the implementation files you created, then immediately begin the next cycle.** The cycle is: implement → run → observe → report → delete → implement → run → observe → report → delete → ... There is no pause between cycles.

**Only reports persist.** Code, scripts, data files — delete them after the report is written. Each cycle starts fresh. This prevents any single implementation choice from dominating future exploration.

## Reports

Reports go in `reports/{language}/` (relative to this file). Each report is a new file: `report-{timestamp}.md`

Create the language subfolder if it does not exist.

### What a report contains

**Pure observation. No interpretation.**

Describe what happened:
- What f was implemented (the rules, not why you chose them)
- What initial state was used
- What M and C were
- What was observed at intervals
- What the final state looked like

Use numbers, counts, descriptions of state. Do not use:
- "emergence"
- "selection"
- "success" / "failure"
- "achieved" / "balanced"
- "interesting" / "promising"
- Any evaluative language

**Wrong:** "Replicators emerged and achieved stable population"
**Right:** "At cycle 1000, 47 locations contained the byte pattern 0x3F. At cycle 2000, 52 locations contained it. At cycle 5000, 48 locations contained it."

**Wrong:** "The system selected for efficient patterns"
**Right:** "Patterns with >4 bits set decreased from 200 to 12. Patterns with <3 bits set increased from 50 to 180."

The interpretation is not yours to make. Report what you observed. Nothing more.

### Previous reports

Before implementing f, read reports from `reports/`:

1. Read the most recent report

Note which physics have been tried. Note which observations recur across reports. Note which physics classes are absent.

Your f should explore physics not yet tried, or vary significantly from prior attempts. Do not repeat what has been tried. The space of possible f is vast — explore it.

**Do not use the same implementation language as the most recent report.** If the last report used Python, use something else. If the last report used JavaScript, use something else. This ensures variety in how f is explored.

Implementation possibilities include: Python, JavaScript, C, Rust, Go, Lua, Bash, AWK, Perl, Ruby, assembly, a custom interpreter, direct memory manipulation, or anything else that can run on this machine.
